TOPP-RA¶

Reachability-Analysis-based Time-Optimal Path Parametrization.

This package produces routines for creation and handling path constraints using the algorithm TOPP-RA.

Submodules¶

toppra.TOPP module¶

The TOPP-RA package currently is implemented using qpOASES.

Interfaces to other solvers will be added in the future.

Notes

Attributes tagged with (qpOASES) are internal variables used with the qpOASES solver. For details on their construction, see belows.

The first nop rows of A are operational rows. They are reserved to specify additional constraints. For example, to specify the constraints that the next stage state must lie inside the target set.

  1. the controllable sets; TODO
  2. the reachable sets; TODO

This class uses qpOASES to solve Quadratic Programs of the form

\[\begin{split}min \quad & 0.5 (u, x, v^T) \mathbf{H} (u, x, v^T)^T + \mathbf{g}^T (u, x, v^T)^T \\ s.t. \quad & \mathbf{l_A}[i] \leq \mathbf{A}[i] (u, x, v^T)^T \leq \mathbf{h_A}[i] \\ & \mathbf{l}[i] \leq (u, x, v^T)^T \leq \mathbf{h}[i]\end{split}\]

The matrices \(A[i], l_A[i], h_A[i]\) consist of four row sections. The four sections are:

  1. operational: see above;
  2. canonical;
  3. non-canonical;
  4. non-canonical hard-bound.

For more details on the last three sections, see PathConstraint’s docstring.

The vectors \(l[i], h[i]\) contain hard-bounds on \(u, x, \mathbf{v}\) respectively.

solve_controllable_sets(eps=1e-14)[source]¶

Solve for controllable sets \(\mathcal{K}_i(I_{\mathrm{goal}})\).

The i-th controllable set \(\mathcal{K}_i(I_{\mathrm{goal}})\) is the set of states at stage \(i\) such that there exists at least a sequence of admissible controls \((u_i,\dots,u_{N-1})\) that drives it to \(\mathcal{I}_{\mathrm{goal}}\).

Notes

The interval computed with one_step is not the true optimal solution. They differ by some small tolerance. This is due to numerical errors with the the QP solver qpOASES. Thus it might happen that the end point of the interval is not actually controllable.

Setting eps to non-zero slight restrict the controllable set, and thus address this problem.

Parameters:eps (float, optional) – A small margin to guard againts accumulating numerical error while computing the one-step sets.
Returns:out – True if \(\mathcal{K}_0(I_{\mathrm{goal}})\) is not empty.
Return type:bool.

  • lG (array, optional) – Shape (N+1, niq). Bounds; Type II.
  • hG (array, optional) – Shape (N+1, niq). Bounds; Type II.
Variables:
  • nm (int) – Dimension of non-canonical inequalities.
  • nv (int) – Dimension of slack variable.
  • neq (int) – Dimension of equality constraint.
  • niq (int) – Dimension of non-canonical inequality constraint.
  • N (int) – Number of discretization segments.

Notes

In the most general setting, a PathConstraint object candescribes one of the following kind:

  1. Canonical:
\[\mathbf a[i] u + \mathbf b[i] x + \mathbf c[i] \leq 0\]
  1. Non-canonical Type I:
\[\begin{split}& \mathbf{abar}[i] u + \mathbf{bbar}[i] x + \mathbf{cbar}[i] = \mathbf{D}[i] v \\ & \mathbf{l}[i] \leq v \leq \mathbf{h}[i]\end{split}\]
  1. Non-canonical Type II, in addition to 2.
\[\mathbf{lG}(s) \leq \mathbf{G}(s) v \leq \mathbf{hG}(s)\]

where u is the path acceleration, x is the squared path velocity and v is a slack variable whose physical meanings depend on the nature of the constraint.

Depending on the constraint, corresponding keyword arguments are to be given as numerical arrays during initialization.

Note that there is no actual problem in allowing a constraint to have both canonical and non-canonical parts.

Example

Create a canonical constraint

>>> a = np.random.randn(N+1, 5)
>>> b = np.random.randn(N+1, 5)
>>> c = np.random.randn(N+1, 5)
>>> pc = PathConstraint(a=a, b=b, c=c, ss=np.linspace(0, 1, N+1))

then, create a first-order interpolation

>>> pc = interpolate_constraint(pc)

toppra.constraints.create_full_contact_path_constraint(path, ss, robot, stance)¶

Contact stability constraint (Colomb frictional model).

Parameters:
  • path (SplineInterpolator)
  • ss (array.) – Shape (N+1, ). Discretization grid points.
  • robot (Pymanoid.Humanoid) – Used for dynamics computation. Torque bounds are taken from the internal OpenRAVE robot.
  • stance (Pymanoid.Stance) – Used for wrench constraint.
Returns:

res – Contact stability constraint.
Return type:

PathConstraint

Note

The dynamics equation of a robot is given by:

\[\mathbf{M}(\mathbf{q}) \ddot{\mathbf{q}}+ \mathbf{C}(\mathbf{q}, \dot{\mathbf{q}})+ \mathbf{g}(\mathbf{q}) = \mathbf{\tau} + \sum_i\mathbf{J}_i(\mathbf{q}, \mathbf{pos_endlink}_i)^T \mathbf{w}_i,\]

where

  • \(\mathbf{q}, \dot{\mathbf{q}}, \ddot{\mathbf{q}}\): robot joint position, velocity and acceleration.
  • \(\tau\): robot joint torque.
  • \(\mathbf{w}_i\): the i-th local contact wrench acting on a link on the robot.
  • \(\mathbf{pos_endlink}_i\): the origin of the wrench w_i.
  • \(\mathbf{J}_i\): the wrench Jacobian at p_i of the link w_i acts on.

According to the linearized Colomb friction model, the sets of feasible contact wrenches are characterized by linear inequalities such as

\[\{\mathbf{w}_i \mid \mathbf{F}_i \mathbf{w}_i \leq 0\}\]

Define the slack variable \(\mathbf{v} := [\tau', \mathbf{w}_1', \mathbf{w}_2', ...]'\) and note that the left-hand side of the dynamic equation can be written in the “abc” form, the path constraint is finally given by

\[\begin{split}& \mathbf{a}(s) u + \mathbf{b}(s) x + \mathbf{c}(s) = \mathbf{D}(s) \mathbf{v} \\ & \mathbf{l}(s) \leq \mathbf{v} \leq \mathbf{h}(s) \\ & \mathbf{l}_G(s) \leq \mathbf{G}(s) \mathbf{v} \leq \mathbf{h}_G(s)\end{split}\]

which is a non-canonical Type II PathConstraint.

Compute the wrench Jacobian for link at point.

We look for J_wrench such that
J_wrench.T * wrench = J_trans.T * F + J_rot.T * tau

return the induced generalized joint torques.

J_wrench is computed by stacking J_translation and J_rotation

toppra.utils.smooth_singularities(pp, us, xs, vs=None)[source]¶

Smooth jitters due to singularities.

Solving TOPP for discrete problem generated from collocation scheme tends to create jitters. This function finds and smooths them.

Notes

(us_smth, xs_smth) is a valid path-parameterization. They satisfy the linear continuity condition \(x_{i+1} = x_i + 2 \Delta_i u_i\).

This function is safe: it will always return a solution.

Parameters:
  • pp (qpOASESPPSolver)
  • us (array) – Shape (_N, ). Controls.
  • xs (array) – Shape (_N+1, ). Squared velocities.
  • vs (array, optional) – If not given, vs_smth will not be returned.
Returns:

  • us_smth (array) – Shape (_N, ). Smoothed controls.
  • xs_smth (array) – Shape (_N+1, ). Smoothed squared velocities.
  • vs_smth (array) – If vs is not given, vs_smth will not be returned.
